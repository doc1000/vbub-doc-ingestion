Document Ingestion Architecture Map

A single-ingestion-contract design for uploaded files, local-path context capture, validation, parsing, normalization, storage, and downstream delivery into search, notes, topic modeling, chunking, and RAG. Built to isolate complexity into small internal APIs.

Single file upload first Original binary preserved Minimal-format text extraction Parser chosen once Audit-friendly boundaries Future Tika fallback possible
Trigger points

Web file picker, drag/drop, browser extension handoff, later URL/webpage ingestion through the same contract.

Core principle

All validation, type detection, extraction, cleanup, and binary prep happen once. Downstream systems consume one normalized payload.

User value

Upload a work document, make notes on it, find it later, and keep a human-readable record of where the original file lived.

Repo strategy

Strong candidate for a separate ingestion repo or service package so it can be tested locally and integrated only after payloads stabilize.

Parser Routing

Choose parser exactly once

Core Contract

ParserRouter

  • Maps canonical MIME/extension to parser adapter
  • Keeps parser logic out of main ingestion flow
  • Optionally routes unknowns to Tika fallback later
Phase 7 — Router Hardening

Hybrid Routing Policy

  • Router no longer relies on suffix alone
  • Strong formats (PDF, DOCX, XLSX, CSV) route by extension first
  • BINARY_REJECT_EXTENSIONS — explicit reject set (.xls, .exe, .dll, .bin, .dat); .xls returns clear 422 recommending .xlsx
  • TEXT_LIKE_EXTENSIONS — aliases (.bat, .jsx, .ps1, .json, .yaml, …) route to TextExtractor only if is_probably_text() confirms content
  • Unknown/suffixless files accepted if is_probably_text() returns True
  • Binary content mislabeled with a text-like suffix is rejected, never reaches TextExtractor

content_classification_service.py

  • Lightweight, stdlib-only helper used exclusively by ParserRouter
  • is_probably_text(file_bytes) — inspects first 8 KB; checks null-byte ratio, UTF-8 decode, printable-character ratio (≥ 0.85)
  • No external text-detection libraries

Core interfaces 
class DocumentExtractor(Protocol):
    def extract(self, file_bytes: bytes, filename: str, canonical_mime: str) -> ExtractionResult: ...

class StorageBackend(Protocol):
    def put(self, *, user_id: str, filename: str, file_bytes: bytes, mime: str) -> BinaryRef: ...

class TextCleaner(Protocol):
    def clean(self, text: str, *, mime: str, hints: dict | None = None) -> CleanTextResult: ...

class TabularTextPolicy(Protocol):
    def filter_rows(self, rows: list[list[str]], *, source_type: str) -> list[list[str]]: ...

class CanonicalDocumentBuilder(Protocol):
    def build(self, *, source: SourceRecord, validation: ValidationResult, binary_ref: BinaryRef,
              extraction: ExtractionResult, clean_text: CleanTextResult, metadata: dict) -> CanonicalDocument: ...
Recommended implementation candidates Lean-first
Stage Primary candidate Why it fits Keep in reserve
API layer FastAPI + Pydantic Typed contracts, easy local dev, strong request/response validation None needed initially
PDF extraction PyMuPDF (fitz) Fast, mature, strong text extraction for general document ingestion pdfplumber for table-heavy edge cases
DOCX extraction python-docx Readable, controllable, Python-native, good for paragraph-oriented extraction docx2txt for ultra-simple text-only fallback
TXT / Markdown stdlib open() + frontmatter-safe text handling Minimal latency and dependency cost markdown-it-py only if heading structure becomes important
CSV extraction stdlib csv module Lightweight, predictable, enough for row filtering and text flattening pandas only if richer tabular inference becomes necessary
XLSX extraction openpyxl Direct workbook access without adding a heavier dataframe dependency pandas for later analysis-oriented workflows
HTML / webpage text BeautifulSoup4 + readability-lxml Good split between DOM parsing and boilerplate/article extraction trafilatura if you later want stronger article extraction
MIME / type detection python-magic Useful for magic-byte sniffing and MIME reconciliation Tika fallback only if format coverage grows
Storage Object/blob storage + thin metadata repo Keeps original binary immutable and separate from extracted text Local filesystem for dev fixtures
Normalization / cleanup Custom shared services Your policy is specific to RAG and note-taking, so custom logic is cleaner than a magic dependency Unstructured/Tika only if you later need broad-format fallback
Package philosophy
  • Prefer the lightest library that exposes the structure you need.
  • Keep pandas out of the critical path unless there is real analytical value.
  • Use custom normalization services for semantic quality instead of outsourcing policy.
  • Add Tika only as a fallback layer, not as the first parser for common formats.

Repo split recommendation + flow chart

Recommended

Separate repo or clearly isolated package

  • Lets you run ingestion locally with fixture documents
  • Keeps parser experiments away from main app
  • Lets Cursor/Codex work on one module at a time
  • Makes payload stability the integration contract
Good first milestone

Definition of done for isolation repo

  • POST endpoint accepts one file
  • Returns CanonicalDocument JSON
  • Stores original binary locally or in object storage
  • Parses at least PDF, DOCX, TXT
  • Fixture-based tests for each parser
Core boundary Good first implementation User-facing or policy detail Shared cleanup logic
Deployment direction Same Fly app Separate process group 
Local dev:
  browser/app -> localhost ingestion FastAPI service

Production:
  same Fly app
    ├── backend process group -> existing backend machine(s)
    └── ingestion process group -> ingestion machine(s)

Call pattern:
  backend -> internal HTTP call -> ingestion service

Benefits:
  - isolated scaling
  - isolated logs and failures
  - simpler than a fully separate Fly app at first
  - same deployment umbrella, but clean process boundary
